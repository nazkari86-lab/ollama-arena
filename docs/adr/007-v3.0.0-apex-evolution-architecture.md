# ADR 007: v3.0.0 "Apex Evolution" Architecture

## Status
Accepted (2026-06-18)

## Context
Ollama-Arena v2.5.0 established a solid foundation with distributed architecture, React frontend, and comprehensive testing. However, the platform was still primarily a **single-turn evaluation tool** limited to prompt-response pairs. 

Key limitations identified:
1. **No Agentic Capabilities**: Models couldn't be evaluated on complex, multi-step tasks
2. **No Learning Loop**: Models didn't improve from arena defeats
3. **Centralized Scaling**: Limited to single-organization deployment
4. **Shallow Metrics**: Only TPS/basic latency, missing energy and hardware insights
5. **Developer Disconnect**: Poor integration into actual development workflows

The v3.0.0 vision addresses these limitations through five major directions:
1. Deep Agentic Evaluation
2. Continuous Auto-Finetuning  
3. Global P2P Grid
4. Deep Hardware & Energy Telemetry
5. "God-Mode" Ecosystem

## Decision

### Architectural Principles

**1. From Evaluation to Evolution**
- Old: Static benchmarking platform
- New: Continuous self-improvement system where models learn from defeats

**2. From Centralized to Decentralized**
- Old: Single-organization deployment
- New: Global P2P network with cryptographic verification

**3. From Shallow to Deep Metrics**
- Old: TPS and basic latency
- New: Energy, memory bandwidth, quantization tradeoffs, hardware profiling

**4. From Tool to Ecosystem**
- Old: Standalone evaluation tool
- New: Integrated into IDE, CI/CD, and development workflows

### Key Architectural Decisions

#### 1. VM Sandbox Isolation
**Decision**: Use KubeVirt/Firecracker for VM sandbox management

**Rationale**:
- KubeVirt integrates with Kubernetes for orchestration
- Firecracker provides lightweight microVMs with minimal overhead
- Strong isolation boundaries for security
- Snapshot/restore capabilities for checkpointing

**Alternatives Considered**:
- Docker containers: Too weak isolation for untrusted code
- Full VMs (VMware/QEMU): Too heavy for rapid spin-up
- WebAssembly: Limited language support and system access

#### 2. P2P Network with Byzantine Consensus
**Decision**: Use libp2p for P2P networking with Byzantine fault-tolerant consensus

**Rationale**:
- libp2p is production-proven (IPFS, Filecoin)
- Byzantine consensus handles malicious nodes
- Reputation system builds trust over time
- Anti-sybil mechanisms prevent node duplication

**Alternatives Considered**:
- Centralized broker: Defeats decentralization goal
- Simple majority consensus: Vulnerable to sybil attacks
- Proof-of-work: Too resource-intensive for evaluation nodes

#### 3. Zero-Knowledge Proofs for Verification
**Decision**: Use ZK-SNARKs for result verification without revealing prompts

**Rationale**:
- Protects both model prompts and benchmark privacy
- Cryptographically proof eliminates fraud claims
- Computationally feasible for result verification
- Vendor protection against false accusations

**Alternatives Considered**:
- Plain cryptographic signatures: No privacy protection
- Trusted execution environments (TEE): Hardware dependencies
- Full transparency: Leaks proprietary benchmarks

#### 4. Auto-Unsloth for Fine-Tuning
**Decision**: Integrate Unsloth for automatic LoRA fine-tuning

**Rationale**:
- Unsloth is optimized for training efficiency
- LoRA adapters are small and fast to train
- Automatic triggering creates continuous improvement loop
- GPU resource queuing maximizes utilization

**Alternatives Considered**:
- Full model fine-tuning: Too resource-intensive
- Manual fine-tuning: Defeats automation goal
- PEFT/LoRA alternatives: Unsloth has better tooling

#### 5. Multi-Platform Hardware Telemetry
**Decision**: Support NVML (NVIDIA), ROCm (AMD), and MPS (Apple) for energy monitoring

**Rationale**:
- Covers major GPU platforms
- Platform-specific optimizations
- Unified abstraction layer for consistency
- Future-proof for new hardware

**Alternatives Considered**:
- NVIDIA-only: Excludes AMD/Apple users
- Vendor-agnostic APIs: Limited functionality
- External power meters: Not practical for distributed deployment

#### 6. WebGL/Three.js for 3D Visualization
**Decision**: Use Three.js with WebGL for neural map visualization

**Rationale**:
- Browser-native, no plugins required
- Hardware-accelerated rendering
- Large ecosystem and community
- React Three Fiber for seamless React integration

**Alternatives Considered**:
- WebGPU: Too early, limited browser support
- Canvas 2D: Limited 3D capabilities
- Server-side rendering: Defeats real-time goal

### Data Flow

**Match Execution Flow**:
1. User requests evaluation (via CLI, Web, or IDE)
2. Task distributed to P2P network (or local execution)
3. Models execute in VM sandboxes with telemetry collection
4. Results cryptographically signed and verified
5. DPO pairs extracted from match outcomes
6. Loser model queued for fine-tuning if threshold met
7. Results anchored to blockchain for immutability
8. Visualization updated in real-time

**Auto-Finetuning Flow**:
1. Model reaches loss threshold (e.g., 1000 defeats)
2. DPO dataset aggregated from recent matches
3. Unsloth training job queued and executed
4. LoRA adapter generated and validated
5. New model variant registered in arena
6. A/B test vs base model for improvement verification
7. If improved, promoted; otherwise, discarded

### Security Model

**Threat Model**:
- Malicious P2P nodes submitting fake results
- Vendors manipulating evaluation results
- Prompt injection in code execution sandboxes
- Data exfiltration through side channels
- Fine-tuning data poisoning

**Mitigations**:
- Byzantine consensus with reputation scoring
- Cryptographic proof with ZKPs
- VM isolation with network restrictions
- Telemetry anomaly detection
- DPO dataset quality filtering

### Performance Considerations

**Scalability**:
- P2P network enables linear scaling
- VM sandbox spin-up time: <30 seconds
- ZK proof verification: <1 second per result
- Fine-tuning jobs: Queued with GPU preemption

**Resource Requirements**:
- Minimum: 8GB RAM, 4 CPU cores for basic features
- Recommended: 16GB RAM, GPU for full features
- P2P node: GPU contribution optional but encouraged

### Backward Compatibility

**Maintained**:
- All v2.5.0 CLI commands
- Existing Jinja2 templates
- REST API endpoints
- Database schema (with migrations)

**New Requirements**:
- Python 3.11+ (for eBPF and async improvements)
- Optional GPU for advanced features
- Optional blockchain node for full verification

### Migration Path

**From v2.5.0**:
1. Update dependencies (pip install)
2. Database migration for new tables
3. Configuration file updates (add new sections)
4. Optional: Enable P2P node participation
5. Optional: Configure auto-finetuning thresholds

## Consequences

### Positive
- **Continuous Improvement**: Models automatically get better through arena defeats
- **Trust**: Cryptographic verification eliminates fraud concerns
- **Scale**: P2P network enables unlimited horizontal scaling
- **Insight**: Hardware telemetry reveals true costs and bottlenecks
- **Integration**: IDE and CI/CD integration brings arena to developers

### Negative
- **Complexity**: Five major directions increase system complexity significantly
- **Resource Requirements**: Some features require specific hardware (GPU, NVML support)
- **Learning Curve**: New paradigms (P2P, ZKPs, fine-tuning) require user education
- **Dependencies**: Additional external dependencies increase maintenance burden

### Risks
- **P2P Network Adoption**: Success depends on critical mass of contributors
- **Fine-Tuning Quality**: Automatic fine-tuning may produce degraded models
- **ZK Proof Performance**: Verification overhead may impact scalability
- **Hardware Compatibility**: Platform-specific telemetry may exclude some users

## Implementation Phases

**Phase 1 (Q3 2026)**: Agentic & Hardware
- VM sandboxes, long-horizon tasks, swarm battles
- Energy telemetry, hardware profiling

**Phase 2 (Q4 2026)**: Swarm & RLHF
- Multi-agent protocol, DPO pipeline
- Auto-Unsloth integration, adversarial generation

**Phase 3 (Q1 2027)**: The Global Grid
- P2P protocol, cryptographic verification
- 3D dashboard, IDE extensions, CI/CD gates
